## 1. Scope of Phase 11

Per the roadmap, Phase 11 deliverables are: **file uploads for labs, document generation panel, PDF output.**

This phase has two halves that share the same Supabase Storage foundation but are otherwise independent:

| Half | Module | Purpose |
|---|---|---|
| A | **AttachmentsModule** | Doctors/Nurses upload lab/imaging files (or type in text results) and attach them to a specific Initial Note or Progress Note. |
| B | **DocumentsModule** | Generate PDF documents (medical certificate, lab request, prescription, charge slip) from current patient/note data, store them, and list them for download. |

Both halves write to Supabase Storage but to different logical concerns — Attachments are **inputs** (evidence uploaded by clinicians), Documents are **outputs** (PDFs generated by the system).

**Explicitly out of scope for Phase 11** (do not implement):
- Audit log writing/interceptor wiring — that's Phase 12. Structure the code so `AuditLogInterceptor` can be dropped in later without refactoring (i.e., don't bypass NestJS's request/response cycle with raw Express handlers).
- OCR triggering for `textResult` — the schema field exists but the OCR pipeline itself is not MVP scope anywhere in the roadmap. Leave `textResult` as a manually-entered field only.

## 3. Authoritative Contracts from the MVP V6 Spec

Do not deviate from these — they are already load-bearing elsewhere in the codebase (Phase 8/9 note forms reference them).

### 3.1 Prisma schema — `documents` table (new model to add)

```prisma
enum DocumentType {
  MEDICAL_CERTIFICATE
  LAB_REQUEST
  PRESCRIPTION
  CHARGE_SLIP
}

model Document {
  id            String       @id @default(uuid()) @db.Uuid
  patientId     String       @map("patient_id") @db.Uuid
  visitId       String?      @map("visit_id") @db.Uuid
  documentType  DocumentType @map("document_type")
  storageKey    String       @map("storage_key")
  generatedBy   String?      @map("generated_by") @db.Uuid
  generatedAt   DateTime     @default(now()) @map("generated_at")

  patient       Patient  @relation(fields: [patientId], references: [id])
  visit         Visit?   @relation(fields: [visitId], references: [id])
  generatedByUser User?  @relation("DocumentGeneratedBy", fields: [generatedBy], references: [id])

  @@map("documents")
}
```

Add the reciprocal `documents Document[]` relation to `Patient` and `Visit` models. Add `NoteType` enum if not already present (`INITIAL_NOTE`, `PROGRESS_NOTE` — should already exist from the `Attachment` model).

### 3.2 Storage flow (Section 3.5 — must match exactly)

1. Client selects file in note editor.
2. Frontend calls `POST /attachments/upload` (multipart form data).
3. NestJS uploads to Supabase Storage bucket **`damayan-attachments`** under path:
   `patients/{patientId}/visits/{visitId}/{filename}`
4. Supabase returns a signed/public URL reference; NestJS persists the `Attachment` record with `storageKey` and `mimeType`.
5. Frontend stores the returned attachment ID in note form state; the note is saved with attachment references.
6. Downloads always go through NestJS-issued **signed URLs** (time-limited), never a public bucket URL.

Documents follow the identical bucket (same `damayan-attachments` bucket, same private-with-signed-URL policy) but a distinct path convention — use `patients/{patientId}/documents/{documentId}.pdf` for clarity and to avoid collisions with attachment filenames.

### 3.3 Bucket provisioning

If not already done in Phase 1–4 setup: create bucket `damayan-attachments` (private, 50MB file size limit) in Supabase. Set bucket policies so authenticated users can upload to their patient folders; all downloads via signed URL only, generated server-side by NestJS with the `service_role` key.

### 3.4 API contract (Section 9.8 / 9.9 — implement exactly these routes)

**Documents**
| Method | Route | Roles |
|---|---|---|
| GET | `/patients/:patientId/documents` | All roles |
| POST | `/patients/:patientId/documents/generate` | Doctor, Nurse, Admin (body: `{ type, visitId? }`) |
| GET | `/patients/:patientId/documents/:id/download` | All roles |

**Attachments**
| Method | Route | Roles |
|---|---|---|
| POST | `/attachments/upload` | Doctor, Nurse, Admin |
| GET | `/attachments/:id` | All roles |
| GET | `/attachments/:id/download` | All roles |
| DELETE | `/attachments/:id` | Doctor, Admin |

### 3.5 RBAC matrix constraint (Section 8.1 — critical, do not soften)

- **Charge slip generation is the one document type Author/Non-Author Doctors cannot generate.** Only Nurse and Admin can call `POST /patients/:patientId/documents/generate` with `type: CHARGE_SLIP`. All other document types (`MEDICAL_CERTIFICATE`, `LAB_REQUEST`, `PRESCRIPTION`) are generatable by Doctor, Nurse, and Admin.
- This means `@Roles()` alone on the controller method is insufficient — the guard/service must branch on the `type` field in the request body. Implement this as a service-level check, not just a decorator, since the decorator can't see the body's `type` value at the route level.

### 3.6 Attachment table constraints (already in schema, respect on service layer)

- `noteType` is `INITIAL_NOTE | PROGRESS_NOTE`, `noteId` references either `initial_notes.id` or `progress_notes.id` — polymorphic, no FK enforced at the DB level for `noteId` itself (only `patientId` has a hard FK). Validate `noteId` belongs to the claimed `noteType` and to the claimed `patientId` at the service layer before persisting.
- `tag` (VARCHAR 100) is the clinician-supplied label (e.g. "CBC", "Chest X-Ray") — this is what `PriorLabsTable` groups by.
- `textResult` is optional and independent of `storageKey` — a clinician can log a lab result as text only, file only, or both. Do not make `storageKey` required at the DTO level if `textResult` is present.

## 4. Backend Implementation

### 4.1 Storage Service (shared by both modules)

Create `backend/src/storage/storage.service.ts` (new shared module — both `AttachmentsService` and `DocumentsService` depend on it, don't duplicate Supabase Storage client logic in each):

```typescript
// backend/src/storage/storage.service.ts
import { Injectable } from '@nestjs/common';
import { SupabaseClient } from '@supabase/supabase-js';

@Injectable()
export class StorageService {
  private readonly bucket = 'damayan-attachments';

  constructor(private readonly supabase: SupabaseClient) {} // injected via SupabaseModule, using service_role key

  async upload(path: string, buffer: Buffer, contentType: string): Promise<string> {
    const { error } = await this.supabase.storage
      .from(this.bucket)
      .upload(path, buffer, { contentType, upsert: false });
    if (error) throw error;
    return path; // this is the storageKey persisted in DB
  }

  async getSignedUrl(storageKey: string, expiresInSeconds = 3600): Promise<string> {
    const { data, error } = await this.supabase.storage
      .from(this.bucket)
      .createSignedUrl(storageKey, expiresInSeconds);
    if (error) throw error;
    return data.signedUrl;
  }

  async delete(storageKey: string): Promise<void> {
    const { error } = await this.supabase.storage.from(this.bucket).remove([storageKey]);
    if (error) throw error;
  }
}
```

Register as a `@Global()` or explicitly exported provider (`StorageModule`) so both `AttachmentsModule` and `DocumentsModule` can import it.

### 4.2 AttachmentsModule

**`backend/src/attachments/dto/create-attachment.dto.ts`**
```typescript
import { IsEnum, IsOptional, IsString, IsUUID, MaxLength } from 'class-validator';
import { NoteType } from '@prisma/client';

export class CreateAttachmentDto {
  @IsUUID() patientId: string;
  @IsEnum(NoteType) noteType: NoteType;
  @IsUUID() noteId: string;
  @IsString() @MaxLength(100) tag: string;
  @IsOptional() @IsString() textResult?: string;
  // file itself arrives via multipart, handled by FileInterceptor, not in the DTO body
}
```

**`backend/src/attachments/attachments.service.ts`** — key methods:
- `upload(dto: CreateAttachmentDto, file: Express.Multer.File | undefined, userId: string)`:
  1. Validate `noteId` exists for the given `noteType` (query `initialNote` or `progressNote` by id, confirm `patientId` matches).
  2. If `file` present: build path `patients/{patientId}/visits/{visitId}/{timestamp}-{originalname}` (fetch `visitId` from the parent note record — both `InitialNote` and `ProgressNote` carry `visitId`), call `storageService.upload(...)`.
  3. Persist `Attachment` row with `storageKey` (or empty if text-only — see note below), `mimeType`, `textResult`, `uploadedBy: userId`.
  4. Return the created attachment (frontend needs the `id` immediately to add to note form state per the storage flow).
- `findByNote(noteType: NoteType, noteId: string)` — used by Initial/Progress Note editors to show already-attached files.
- `findByPatient(patientId: string)` — used by `PriorLabsTable`, grouped by `tag` in the service or left ungrouped and grouped client-side (prefer service-side grouping to keep the frontend dumb: return `{ tag, attachments: [...] }[]`).
- `getDownloadUrl(id: string)` — fetch `storageKey`, delegate to `storageService.getSignedUrl`.
- `remove(id: string)` — delete storage object (if `storageKey` present) then DB row. Role-gated to Doctor/Admin at the controller.

If `storageKey` must remain non-nullable per the current schema (it says `NOT NULL` in Section 6.2), and a text-only attachment is allowed, either (a) migrate the column to nullable, or (b) store a sentinel/no-op key. **Prefer (a)** — the MVP V6 attachment table note explicitly says `textResult` supports "Optional text entry alongside or instead of file," which is inconsistent with `storage_key NOT NULL`. Flag this as a schema correction: run a migration making `storage_key` nullable before implementing the text-only path.

**`backend/src/attachments/attachments.controller.ts`** — implement the 4 routes from Section 3.4 exactly, with `@UseGuards(JwtAuthGuard, RolesGuard)` and per-route `@Roles(...)`. Use `@UseInterceptors(FileInterceptor('file'))` on the upload route, memory storage (buffer passed straight to `StorageService.upload`, no temp disk write).

### 4.3 DocumentsModule

**`backend/src/documents/documents.service.ts`** — key method `generate(patientId, type, visitId, userId)`:
1. RBAC check: if `type === 'CHARGE_SLIP'`, verify caller's role is Nurse or Admin (throw `ForbiddenException` otherwise). Do this in the service, not just the controller decorator, per Section 3.5 above.
2. Fetch the data needed for the requested template:
   - `MEDICAL_CERTIFICATE`: patient demographics, latest published note's assessment/diagnosis.
   - `LAB_REQUEST`: patient demographics, diagnostics array from the active note (`initial_notes.diagnostics` or `progress_notes.diagnostics` JSONB).
   - `PRESCRIPTION`: patient demographics, active medications (`medications` where `isActive: true`).
   - `CHARGE_SLIP`: patient demographics, visit info, services/problems addressed during the visit.
3. Render PDF via `pdfkit` (lighter weight, no headless Chromium dependency — prefer this over `puppeteer` unless the templates need complex CSS layout the team already has in HTML form).
4. Upload buffer to Supabase Storage at `patients/{patientId}/documents/{uuid}.pdf`.
5. Persist `Document` row (`documentType`, `storageKey`, `generatedBy: userId`, `visitId`).
6. Return the created `Document` record.

**`backend/src/documents/templates/`** — one render function per document type (`medical-certificate.template.ts`, `lab-request.template.ts`, `prescription.template.ts`, `charge-slip.template.ts`), each taking the fetched data and a `PDFDocument` instance, so `DocumentsService.generate` stays a thin orchestrator and templates stay independently testable.

**`backend/src/documents/documents.controller.ts`** — implement the 3 routes from Section 3.4. `POST /generate` has `@Roles('DOCTOR', 'NURSE', 'ADMIN')` at the decorator level (broad gate) with the charge-slip-specific narrowing done inside the service as described above.

## 5. Frontend Implementation

### 5.1 `useAttachments.ts` hook

```typescript
// frontend/src/hooks/useAttachments.ts
export function useAttachmentsByNote(noteType: 'INITIAL_NOTE' | 'PROGRESS_NOTE', noteId: string | undefined) {
  return useQuery({
    queryKey: ['attachments', noteType, noteId],
    queryFn: () => api.get(`/attachments?noteType=${noteType}&noteId=${noteId}`),
    enabled: !!noteId, // see 5.4 for why noteId can be undefined
  });
}

export function useUploadAttachment() {
  const queryClient = useQueryClient();
  return useMutation({
    mutationFn: (params: { file?: File; patientId: string; noteType: string; noteId: string; tag: string; textResult?: string }) => {
      const formData = new FormData();
      if (params.file) formData.append('file', params.file);
      formData.append('patientId', params.patientId);
      formData.append('noteType', params.noteType);
      formData.append('noteId', params.noteId);
      formData.append('tag', params.tag);
      if (params.textResult) formData.append('textResult', params.textResult);
      return api.post('/attachments/upload', formData, {
        onUploadProgress: (e) => { /* feed into UploadProgressBar percent prop */ },
      });
    },
    onSuccess: (_, vars) => queryClient.invalidateQueries({ queryKey: ['attachments', vars.noteType, vars.noteId] }),
  });
}

export function usePriorLabs(patientId: string) {
  return useQuery({
    queryKey: ['attachments', 'patient', patientId],
    queryFn: () => api.get(`/attachments?patientId=${patientId}`), // grouped by tag server-side per 4.2
  });
}

export function useDeleteAttachment() { /* mutation, Doctor/Admin only — hide the control client-side too */ }
```

### 5.2 The noteId-before-upload problem (resolve using the existing auto-save architecture)

`attachments_context.md` flags this as an open caveat. **Resolution: don't invent a new flow — reuse the auto-save pattern already specified for Phase 8.** Section 11.3 already states the API integration is `POST /initial-note on first save; PATCH on subsequent`. That means:

1. On mount of `InitialNoteForm` (or `ProgressNoteForm`), if no `noteId` exists yet (new note), immediately fire the `POST` to create a draft-status note record — even before the clinician has typed anything meaningful (chief complaint can be an empty string that gets validated only at publish time, not at draft-creation time).
2. The returned `noteId` is stored in local form state and used for every subsequent `PATCH` (existing auto-save behavior) **and** now also for `AttachmentsSection`'s `noteId` prop.
3. `AttachmentsSection` renders in a disabled/"save a draft first" state only in the theoretical instant between component mount and the first `POST` resolving — this window is sub-second and can be covered by `LabResultsSectionSkeleton.tsx` (already built, exactly for this purpose).

This keeps attachment linking consistent with how the rest of the note form already works and avoids introducing client-side-only "pending" attachment state that would need reconciliation logic later.
